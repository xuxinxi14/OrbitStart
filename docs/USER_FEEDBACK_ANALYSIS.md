## 一、简单问题（P0–P1，可快速修复）

### 问题 1：弹窗无法点击空白处关闭

**现象**：设置、插件等弹窗打开后，点击弹窗外的空白区域不会关闭弹窗。

**根因**：`palette-backdrop` 容器（App.tsx 第 2480/2521/2555/2582/2628/3505 行）只作为定位容器，没有绑定 `onClick` 背景关闭逻辑。只有右上角 × 按钮能关闭。

**解决方案**：
- 给每个 `palette-backdrop centered-backdrop` 的 `<section>` 加 `onClick`，判断 `event.target === event.currentTarget`（点击的是背景本身而非内部内容）时调用关闭。
- 封装一个 `<ModalBackdrop onClose={...}>` 组件统一处理，避免每个弹窗重复写。
- 注意：命令面板（palette）不宜点击空白关闭（用户可能误触），需单独排除或保留 Esc 关闭。

**工作量**：小（1–2 小时）。

### 问题 2：弹窗层级错误（新弹窗出现在下层）

**现象**：在某个弹窗内点击"设置""插件"等按钮，新弹窗出现在当前弹窗的下层（被遮挡）。

**根因**：层级管理不统一。
- 设置弹窗 `localAuxPanel` 用 `style={{ zIndex: 100 }}`（App.tsx 第 3505 行）。
- TripPanel 用 `palette-backdrop`，其 z-index 在 CSS 中定义为 `9999 !important`（styles.css 第 2825 行）。
- 命令面板同样用 `palette-backdrop`。
- 多个弹窗用同一 class，后渲染的未必在上层；`localAuxPanel` 的 100 又低于 `palette-backdrop` 的 9999，导致从命令面板打开设置时设置被压在下层。

**解决方案**：
- 建立统一的 z-index 层级表（CSS 变量或常量），例如：
  - `--z-base: 1`
  - `--z-dropdown: 50`
  - `--z-context-menu: 80`
  - `--z-modal-backdrop: 90`（TripPanel / 编辑器）
  - `--z-aux-panel: 95`（设置/插件/主题弹窗）
  - `--z-command-palette: 9999`（命令面板，最高）
- 弹窗嵌套时，后打开的自动 +1 或使用递增计数器。
- 移除 `localAuxPanel` 的硬编码 `zIndex: 100`，改用统一变量。

**工作量**：小（2–3 小时，含全量排查）。

### 问题 3：新建分组无法删除

**现象**：用户新建分组后，没有任何入口可以删除该分组。

**根因**：后端只有 `create_group` 命令（main.rs 第 3411 行），**没有 `delete_group` 命令**。前端也没有删除分组的 UI 入口。`OrbitGroup` 有 `custom` 字段标记自定义分组，但删除逻辑未实现。

**解决方案**：
1. **后端**：新增 `delete_group` Tauri 命令：
   - 校验 `custom == true`（默认分组不可删）。
   - 将该分组下的 items 迁移到默认分组（或要求用户先清空）。
   - 删除 `groups` 表记录。
2. **前端**：两个入口：
   - 分组栏右键 → "删除分组"（依赖问题 6 的右键菜单改造）。
   - 设置页 → 引擎管理/分组管理 → 分组列表的删除按钮。
3. 删除前弹确认框，提示"该分组下有 N 个资源将迁移到默认分组"。

**工作量**：中（半天）。

### 问题 4：点击卡片后位置频繁移动

**现象**：点击资源卡（应用、网页、PS 等）后，卡片立即移动到前排。频繁移动图标位置造成用户"强烈的不安感和不适感"。

**根因**：资源中心空查询时的排序逻辑（searchEngine.ts 第 403–426 行）按 `recencyBonus(lastLaunchedAt) → launchCount → favorite → title` 排序。点击卡片会调用 `launchItem`，后端更新 `last_launched_at`（main.rs 第 2668 行）和 `launch_count`，前端刷新后卡片因 recency 变高而前移。

这是之前为修复"只显示收藏"bug 时优化的排序——"最近使用优先"。但对资源中心主视图而言，位置不稳定是反直觉的。

**解决方案**（推荐方案 A）：
- **方案 A（推荐）**：资源中心主视图默认使用**稳定排序**（手动顺序 / 字母序 / 分组），`recency` 排序**只在命令面板（palette）中生效**。这样：
  - 资源中心 = 稳定的"我的工作台"，位置固定。
  - 命令面板 = 动态的"最近使用快速访问"，recency 在此有价值。
- 方案 B：资源中心增加"锁定排序"开关，用户自选是否启用 recency 排序。
- 方案 C：点击不立即重排，延迟到下次切换视图时才重排（治标不治本）。

**推荐**：方案 A。需要给 `OrbitItem` 增加一个 `sortOrder` 字段（手动排序）或直接按 `title` 字母序，资源中心用稳定排序，`buildSortedResults` 的空查询分支在"主视图"和"命令面板"两种调用场景下用不同排序策略。

**工作量**：中（半天，含排序策略重构）。

### 问题 5：tooltip 响应慢、外观差

**现象**：图标按钮悬浮后说明文字响应过慢，甚至无响应；文字位置和外观不舒服。

**根因**：全站使用原生 HTML `title` 属性作为 tooltip（App.tsx 中约 20+ 处 `title=`）。原生 title tooltip 的固有缺陷：
1. 浏览器默认延迟约 500ms–1s，且不可配置。
2. 样式由操作系统决定，无法自定义（字体、颜色、位置）。
3. 不会立即跟随鼠标移动。
4. 在 Tauri WebView 中表现更不稳定（有时不显示）。

**解决方案**：
- 实现一个轻量自定义 Tooltip 组件（基于 CSS + 少量 JS）：
  - `onMouseEnter` 触发，延迟可配（建议 200ms）。
  - 用 CSS 定位（absolute），支持 top/bottom/left/right。
  - 样式与主题一致（`var(--surface)` / `var(--text)` / `var(--shadow)`）。
  - 支持主题适配（所有 `data-theme` 自动继承）。
- 将全站 `title="xxx"` 替换为 `<Tooltip content="xxx">`。
- 可用 Alpine.js 或纯 React state 实现，无需引入新依赖。

**工作量**：中（组件 1–2 小时 + 全站替换 2–3 小时）。

### 问题 6：右键菜单全局单一，无分组栏菜单

**现象**：软件全局只有一个右键菜单，分组栏右键无法提供"删除分组"等专属功能。

**根因**：`ContextMenuKind = "resource" | "blank" | "edit"`（contextMenu.ts 第 1 行），只有三种类型。`contextMenuFromEvent` 根据点击目标判断：可编辑元素 → edit，有 `data-resource-id` → resource，其余 → blank。分组栏没有对应的 kind，右键只会落到 "blank"。

**解决方案**：
1. **扩展 ContextMenuKind**：新增 `"group"` 类型，携带 `groupId`。
2. **contextMenu.ts**：
   - `resourceIdFromTarget` 旁新增 `groupIdFromTarget`（查找 `[data-group-id]`）。
   - `contextMenuFromEvent` 增加 group 分支。
3. **App.tsx renderContextMenu**：新增 `contextMenu.kind === "group"` 分支，渲染：
   - 重命名分组
   - 删除分组（依赖问题 3）
   - （可选）设为默认分组
4. **分组栏 DOM**：给分组元素加 `data-group-id` 属性。
5. 可扩展更多区域化右键菜单（如侧栏、KPI 卡片），后续按需加 kind。

**工作量**：中（半天，与问题 3 合并实施更高效）。

## 二、其他方向（P2，需系统性梳理）

### 问题 7：内容重复、信息混乱、命名图标矛盾

**现象**：重复内容过多；命名和图标不常见且前后矛盾。

**根因分析**（未深入审计，建议后续专项梳理）：
- `src/data/catalog.ts` 的默认资源（12 项）可能存在重复或命名不一致。
- 图标用 Lucide 图标名（字符串），可能存在"同一功能用不同图标"或"不同功能用相似图标"。
- 命名可能中英混用、术语不统一。

**解决方案**：
1. **内容审计**：导出当前所有默认资源 + 用户资源，按"标题/目标/图标"去重对比。
2. **命名规范**：制定命名约定（如"应用用中文名、网址用品牌名、脚本用功能名"），全量过一遍 catalog.ts。
3. **图标规范**：建立图标映射表（kind → 推荐图标），如 app→AppWindow、website→Globe、script→TerminalSquare、file→FileText，确保同类资源图标一致。
4. **去重工具**：在设置页加一个"查重"入口，扫描 title/target 相似的资源，让用户批量清理。

**工作量**：中（内容审计 1 天 + 规范制定 0.5 天 + 落地 0.5 天）。

### 问题 8：操作不直观且繁琐

**现象**：常见操作路径过长或不明显。

**根因分析**（需结合具体操作场景进一步确认）：
- 添加资源可能需要多步（选类型 → 填表单 → 选图标 → 选分组）。
- 编辑资源需右键 → 编辑 → 弹窗 → 保存，路径偏长。
- 批量操作入口可能不显眼。

**解决方案**：
1. **梳理核心操作路径**，列出"添加/编辑/删除/搜索/启动/分组"每个操作的点击次数，目标是 ≤2 次点击完成核心操作。
2. **拖拽创建**已有（`createItemsFromPaths`），但可能入口不明显，需在 UI 上给出拖拽提示。
3. **快捷键**：为核心操作绑定快捷键（如 Ctrl+N 新建、Ctrl+F 搜索、Ctrl+E 编辑）。
4. **减少弹窗层级**：能内联编辑的不弹窗（如分组重命名直接在分组栏内联输入）。

**工作量**：中（需先做操作路径审计，再逐项优化）。

### 问题 9："工作区/快捷打开"定位不突出

**现象**：软件的主要功能"工作区"和"快捷打开"的特点不够突出。

**根因分析**：
- 当前 dashboard（renderDashboard，App.tsx 第 1495 行）是 KPI 概览 + 资源网格，偏"管理后台"风格，不像"启动工作台"。
- "工作区"概念（action_chain）在 UI 上没有特殊入口或视觉强调。
- "快捷打开"的核心价值（命令面板 + 全局热键）入口在侧栏底部，不够突出。

**解决方案**：
1. **Dashboard 重设计**：首屏应该是"快速启动区"——最大的搜索/命令面板入口 + 最近使用的 6–8 个资源大卡片，而非 KPI 数字。KPI 可以缩小到角落。
2. **工作区入口前置**：侧栏"工作区"（action_chain）作为独立一级入口，而非混在资源列表里。
3. **命令面板 CTA**：dashboard 顶部放一个大的"按 Ctrl+Alt+Space 快速启动"提示卡片，引导用户使用核心能力。
4. **全局热键可见性**：在 topbar 常驻显示当前热键（已有 `themeLabel · {globalHotkey}`，但可更醒目）。

**工作量**：大（需产品重新设计 dashboard 布局，1–2 天）。
