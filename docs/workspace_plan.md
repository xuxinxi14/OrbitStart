\# OrbitStart 工作区插件计划书



\## 一、功能定位



\### 1.1 插件名称



建议名称：



\* 中文：\*\*工作区插件\*\*

\* 英文：\*\*Workspace Plugin\*\*

\* 备用名称：启动场景插件 / Launch Scene Plugin / Orbit Workspace



\### 1.2 插件定位



该功能不应作为 OrbitStart 主程序的核心功能直接内置，而应作为一个独立插件实现。



它的核心能力是：



> 将一个项目或使用场景所需的应用、网址、文件夹、文件、脚本和命令保存成一个“工作区”，并支持一键按顺序启动。



例如：



```text

工作区：OrbitStart 开发

\- VS Code

\- E:\\OrbitStart 项目文件夹

\- GitHub 仓库

\- Tauri 文档

\- 本地终端命令

```



用户点击一次即可启动整个项目环境。



\---



\## 二、为什么必须做成插件



\### 2.1 避免主程序臃肿



工作区功能后期会涉及：



\* 启动顺序；

\* 启动参数；

\* 工作目录；

\* 延迟启动；

\* 脚本执行；

\* 条件等待；

\* 窗口位置恢复；

\* 多显示器布局；

\* 进程检测；

\* 端口检测。



这些能力会让功能复杂度迅速膨胀。如果直接塞进 OrbitStart 主体，会让主程序越来越重。



因此更合理的定位是：



> OrbitStart 主体负责资源组织、搜索、启动和插件承载；

> 工作区插件负责复杂场景启动和工作环境恢复。



\### 2.2 保持 OrbitStart 核心定位清晰



OrbitStart 的核心定位应该保持为：



> 本地资源启动工作台。



工作区插件则是一个高级扩展：



> 项目工作环境启动器。



这样可以避免产品定位混乱。



\### 2.3 方便用户选择



不是所有用户都需要工作区功能。



有些用户只需要：



\* 应用启动；

\* 网站收藏；

\* 简单分组；

\* Command Bar。



有些高级用户才需要：



\* 一键启动开发环境；

\* 延迟启动；

\* 脚本；

\* 窗口布局恢复。



插件化后，用户可以按需启用，不需要的人不会被复杂功能干扰。



\### 2.4 方便后续维护



工作区功能涉及系统能力较多，后期容易出 bug。插件化可以让它：



\* 单独迭代；

\* 单独测试；

\* 单独开关；

\* 单独禁用；

\* 不影响 OrbitStart 主程序稳定性。



\---



\## 三、插件与主程序的关系



\### 3.1 主程序提供的能力



OrbitStart 主程序应向插件提供这些基础能力：



1\. 读取已有资源条目；

2\. 启动应用；

3\. 打开网址；

4\. 打开文件夹；

5\. 打开文件；

6\. 访问基础配置；

7\. 注册插件页面；

8\. 注册 Command Bar 搜索结果；

9\. 注册插件快捷入口；

10\. 显示 toast / 通知 / 错误提示。



\### 3.2 工作区插件负责的能力



工作区插件自己负责：



1\. 工作区列表；

2\. 工作区编辑器；

3\. 工作区步骤管理；

4\. 启动顺序；

5\. 延迟启动；

6\. 启动参数；

7\. 工作目录；

8\. 启动日志；

9\. 条件等待；

10\. 窗口布局保存与恢复。



\### 3.3 插件不应该做的事情



工作区插件不应直接接管 OrbitStart 主体功能，例如：



\* 不替代主界面分组；

\* 不替代普通应用启动；

\* 不替代 Command Bar；

\* 不强制改变现有 item / group 模型；

\* 不默认扫描系统全部程序；

\* 不默认执行危险脚本；

\* 不默认捕获用户所有窗口。

\## 四、插件功能总路线



开发路线分为 6 个阶段：



```text

Phase 0：插件可行性评估

Phase 1：工作区插件 MVP

Phase 2：启动参数、工作目录、延迟启动

Phase 3：插件体验增强与 Command Bar 集成

Phase 4：脚本、条件等待、依赖顺序

Phase 5：窗口位置保存与恢复

Phase 6：接近 PowerToys Workspaces 的完整体验

```



\---



\# Phase 0：插件可行性评估



\## 目标



确认 OrbitStart 当前插件系统是否能承载工作区功能。



\## 需要检查



1\. 当前插件系统是否支持独立页面；

2\. 插件是否可以读取 catalog\_snapshot；

3\. 插件是否可以调用 launch\_item 或等价启动能力；

4\. 插件是否可以保存自己的数据；

5\. 插件是否有独立配置区；

6\. 插件是否可以向 Command Bar 注册结果；

7\. 插件是否可以显示通知和错误；

8\. 插件是否可以调用 Tauri 后端命令；

9\. 插件是否支持权限声明；

10\. 插件是否可以被禁用而不影响主程序。



\## Phase 0 输出



Codex 应输出评估报告，说明：



\* 现有插件 API 是否足够；

\* 需要补哪些插件能力；

\* 哪些能力应由主程序提供；

\* 哪些能力应由插件内部实现；

\* 最小 MVP 需要改哪些文件；

\* 是否需要新增插件权限模型。



\---



\# Phase 1：工作区插件 MVP



\## 目标



实现一个最小可用插件：



> 用户可以创建工作区，从已有 OrbitStart 资源中选择应用、网站、文件夹加入工作区，然后一键按顺序启动。



\## 必做功能



1\. 新增工作区插件入口；

2\. 插件拥有独立页面；

3\. 支持创建工作区；

4\. 支持编辑工作区名称、描述、图标；

5\. 支持从已有资源中添加启动项；

6\. 支持调整启动顺序；

7\. 支持启用 / 禁用某个启动项；

8\. 支持一键启动整个工作区；

9\. 按顺序启动所有步骤；

10\. 显示启动成功 / 失败结果。



\## 暂不做功能



第一版不做：



\* 脚本执行；

\* 条件等待；

\* 端口检测；

\* 进程检测；

\* 窗口位置恢复；

\* 自动捕获桌面状态；

\* 多显示器布局恢复；

\* 复杂依赖管理。



\## MVP 数据结构



```ts

type Workspace = {

&#x20; name: string

}

&#x20; id: string

&#x20; itemId?: string

&#x20; title: string

&#x20; target: string

&#x20; enabled: boolean

&#x20; delayMs?: number

&#x20; createdAt: string

&#x20; updatedAt: string

}

```



\## MVP 验收标准



1\. 插件可以独立启用；

2\. 插件可以独立关闭；

3\. 可以创建工作区；

4\. 可以添加已有资源；

5\. 可以调整顺序；

6\. 可以一键启动；

7\. 某个启动项失败不影响后续步骤；

8\. 不破坏主程序原有分组、搜索、Command Bar；

9\. 不污染主程序核心数据结构。

\# Phase 2：启动参数、工作目录、延迟启动



\## 目标



让工作区插件支持更真实的开发场景。



\## 新增能力



\### 2.1 启动参数



支持给应用添加参数，例如：



```text

code.exe E:\\Projects\\OrbitStart

chrome.exe --profile-directory="Profile 2"

game.exe -windowed -dx11

```



\### 2.2 工作目录



支持配置 working directory，例如：



```text

E:\\Projects\\OrbitStart

```



适合：



\* 开发项目；

\* 命令行工具；

\* 便携软件；

\* 脚本启动。



\### 2.3 延迟启动



支持每个步骤设置延迟：



```text

第 1 步：启动 VMware

第 2 步：等待 30 秒

第 3 步：打开 Samba 目录

```



\### 2.4 失败策略



每个步骤支持：



\* 失败后继续；

\* 失败后停止；

\* 失败后询问用户；

\* 失败后重试。



\## 扩展数据结构



```ts

type WorkspaceStep = {

&#x20; id: string

&#x20; workspaceId: string

&#x20; order: number

&#x20; type: "item" | "app" | "website" | "folder" | "file" | "command"

&#x20; title: string

&#x20; target: string

&#x20; args?: string\[]

&#x20; workingDirectory?: string

&#x20; enabled: boolean

&#x20; delayMs?: number

&#x20; failurePolicy: "continue" | "stop" | "ask" | "retry"

&#x20; retryCount?: number

&#x20; createdAt: string

&#x20; updatedAt: string

}

```



\## 安全边界



这一阶段仍然不建议开放完全自由的 shell 命令。



推荐先支持：



```text

程序路径 + 参数 + 工作目录

```



暂时谨慎支持：



```text

cmd /c

powershell

bat

ps1

```

\# Phase 3：插件体验增强与 Command Bar 集成



\## 目标



让工作区插件更好用，并融入 OrbitStart 的整体体验。



\## 新增功能



\### 3.1 Command Bar 集成



工作区插件可以向 Command Bar 注册结果：



```text

启动工作区：OrbitStart 开发

启动工作区：论文写作

启动工作区：视频剪辑

```



用户搜索工作区名称后，回车即可启动。



\### 3.2 首页卡片入口



插件可以向 OrbitStart 首页注册卡片：



\* 常用工作区；

\* 最近启动工作区；

\* 收藏工作区。



\### 3.3 从分组创建工作区



用户可以把已有分组转换为工作区：



```text

开发区 → 创建为工作区

```



\### 3.4 启动日志



记录每次工作区启动：



\* 启动时间；

\* 启动项数量；

\* 成功数量；

\* 失败数量；

\* 总耗时；

\* 错误原因。



\### 3.5 启动进度面板



启动时显示：



```text

正在启动：2 / 6

当前步骤：VS Code

等待：30 秒后打开 Samba 目录

失败：0

```



\## 验收标准



1\. Command Bar 可以搜索并启动工作区；

2\. 首页可以展示常用工作区；

3\. 可以从分组生成工作区；

4\. 可以查看启动日志；

5\. 启动失败有明确提示；

6\. 插件禁用后，相关入口自动消失。

\# Phase 4：脚本、条件等待与依赖顺序



\## 目标



支持更复杂的真实工作流。



\## 新增功能



\### 4.1 脚本步骤



支持添加：



\* `.bat`

\* `.cmd`

\* `.ps1`

\* 后续支持 `.sh`



脚本功能必须有明确风险提示：



```text

脚本会在本机执行，请只运行你信任的脚本。

```



\### 4.2 条件等待



支持等待条件：



| 类型      | 示例                |

| ------- | ----------------- |

| 固定时间    | 等待 30 秒           |

| 进程出现    | 等待 vmware.exe 启动  |

| 端口可访问   | 等待 localhost:3000 |

| 路径存在    | 等待 Samba 目录挂载     |

| URL 可访问 | 等待本地服务启动          |



\### 4.3 依赖顺序



允许步骤之间设置依赖关系：



```text

步骤 3 必须等待步骤 1 成功后执行

步骤 4 必须等待 localhost:3000 可访问后执行

```



\## 扩展数据结构



```ts

type WorkspaceStep = {

&#x20; id: string

&#x20; workspaceId: string

&#x20; order: number

&#x20; type: "app" | "website" | "folder" | "file" | "command" | "script" | "wait"

&#x20; title: string

&#x20; target?: string

&#x20; args?: string\[]

&#x20; workingDirectory?: string

&#x20; enabled: boolean

&#x20; delayMs?: number

&#x20; waitCondition?: {

&#x20;   type: "time" | "process" | "port" | "path" | "url"

&#x20;   value: string

&#x20;   timeoutMs?: number

&#x20; }

&#x20; dependsOn?: string\[]

&#x20; failurePolicy: "continue" | "stop" | "ask" | "retry"

&#x20; retryCount?: number

}

```



\## 风险控制



1\. 默认不开启脚本能力；

2\. 第一次使用脚本时弹出说明；

3\. 不允许插件静默执行未知脚本；

4\. 工作区导入时应提示脚本风险；

5\. 后续可加入“安全模式”，禁用所有脚本步骤。

\# Phase 5：窗口位置保存与恢复



\## 目标



开始向 PowerToys Workspaces 的核心能力靠近：



> 保存当前应用窗口位置，下次启动时尽量恢复。



\## 插件化原则



窗口恢复属于高风险、高复杂度能力，必须放在工作区插件内部，不进入主程序核心。



主程序最多提供底层 Tauri 命令或权限接口，具体逻辑由插件控制。



\## 第一版窗口能力



\### 5.1 手动保存窗口布局



用户打开某个工作区后，点击：



```text

保存当前窗口布局

```



插件记录：



\* 进程名；

\* 窗口标题；

\* 可执行文件路径；

\* 窗口坐标；

\* 窗口大小；

\* 是否最大化；

\* 显示器信息。



\### 5.2 启动后尝试恢复窗口



执行工作区时：



1\. 启动应用；

2\. 等待窗口出现；

3\. 匹配窗口；

4\. 移动窗口；

5\. 调整大小；

6\. 恢复最大化状态。



\### 5.3 恢复失败提示



例如：



```text

已打开应用，但未能恢复窗口位置。

可能原因：窗口标题变化、应用启动较慢、权限不足。

```



\## 数据结构



```ts

type WorkspaceWindowLayout = {

&#x20; id: string

&#x20; workspaceId: string

&#x20; stepId?: string

&#x20; processName: string

&#x20; windowTitle?: string

&#x20; executablePath?: string

&#x20; x: number

&#x20; y: number

&#x20; width: number

&#x20; height: number

&#x20; monitorId?: string

&#x20; isMaximized?: boolean

&#x20; isMinimized?: boolean

&#x20; capturedAt: string

}

```



\## 阶段边界



这一阶段只建议支持：



\* Windows；

\* 普通桌面窗口；

\* 单显示器优先；

\* 不保证所有软件都能恢复；

\* 不支持复杂虚拟桌面；

\* 不强行移动权限不足的窗口。

\# Phase 6：完整 PowerToys Workspaces 级体验



\## 目标



实现接近 PowerToys Workspaces 的完整工作区体验，但仍保持 OrbitStart 的插件化边界。



\## 完整能力



\### 6.1 捕获当前工作区



用户点击：



```text

捕获当前工作区

```



插件扫描当前窗口，并生成工作区草稿。



\### 6.2 选择保存窗口



捕获后显示：



```text

\[✓] VS Code

\[✓] Chrome - GitHub

\[✓] Obsidian

\[ ] 微信

\[ ] QQ

```



用户选择哪些窗口要保存。



\### 6.3 自动生成启动步骤



插件根据捕获结果生成：



\* 应用路径；

\* 启动参数；

\* 窗口标题；

\* 窗口位置；

\* 窗口大小。



\### 6.4 一键恢复完整工作区



执行时：



1\. 启动应用；

2\. 打开网页；

3\. 打开文件夹；

4\. 执行脚本；

5\. 等待条件；

6\. 恢复窗口位置；

7\. 恢复窗口大小；

8\. 处理失败步骤。



\### 6.5 多显示器支持



记录：



\* 显示器 ID；

\* 分辨率；

\* 缩放比例；

\* 显示器相对位置；

\* 主显示器。



当显示器布局变化时，提供 fallback：



\* 移动到主显示器；

\* 按比例缩放位置；

\* 提示用户重新保存布局。



\### 6.6 工作区快捷键



用户可以为工作区绑定快捷键：



```text

Ctrl + Alt + 1：启动开发工作区

Ctrl + Alt + 2：启动写作工作区

```

\## 五、插件权限设计



工作区插件需要权限分级，避免一开始就拥有过多系统能力。



\### 基础权限



Phase 1 到 Phase 3 需要：



\* 读取 OrbitStart 已添加资源；

\* 打开应用；

\* 打开网址；

\* 打开文件夹；

\* 保存插件数据；

\* 注册插件页面；

\* 注册 Command Bar 结果。



\### 高级权限



Phase 4 之后需要：



\* 执行脚本；

\* 检测进程；

\* 检测端口；

\* 检测路径；

\* 读取窗口列表；

\* 移动窗口；

\* 调整窗口大小。



\### 权限提示



首次启用高级功能时应提示：



```text

工作区插件需要更高权限来执行脚本、检测进程或恢复窗口位置。请确认你信任当前工作区配置。

```

\## 六、插件数据存储设计



工作区插件的数据应尽量独立，避免污染主数据库核心表。



推荐方式：



```text

plugin\_data/workspace-plugin/

```



或数据库中使用插件命名空间：



```text

plugin\_workspace\_scenes

plugin\_workspace\_steps

plugin\_workspace\_logs

plugin\_workspace\_window\_layouts

```



\### 推荐表结构



```text

plugin\_workspace\_scenes

plugin\_workspace\_steps

plugin\_workspace\_logs

plugin\_workspace\_window\_layouts

```



这样即使插件被禁用或卸载，也不会破坏 OrbitStart 主体数据。

\* 古金色强调；

\* 卡片式工作区；

\* 启动轨道感动效。

\## 八、与 Command Bar 的关系



Command Bar 不应直接内置工作区逻辑，而应允许插件注册结果。



工作区插件注册：



```text

类型：workspace

标题：启动工作区：OrbitStart 开发

动作：launchWorkspace(workspaceId)

```



Command Bar 只负责展示和触发，不负责管理工作区内部逻辑。



这样可以保持边界清晰：



| 模块          | 负责内容             |

| ----------- | ---------------- |

| Command Bar | 搜索、展示、触发         |

| 工作区插件       | 工作区数据、执行逻辑、日志    |

| 主程序         | 基础启动能力、插件 API、权限 |

```text

请对 OrbitStart 当前代码库进行只读评估，判断是否适合以“插件”的形式新增 Workspace Plugin 工作区插件。



目标插件：

用户可以创建一个工作区，把多个已有应用、网站、文件夹、文件加入工作区，并一键按顺序启动。后续逐步支持启动参数、工作目录、延迟启动、脚本、条件等待和窗口位置恢复。
```

\## 十一、最终结论



工作区功能值得做，但必须插件化。



它不应该成为 OrbitStart 主体的一部分，而应该作为一个高级插件存在。



最合理的产品结构是：



```text

OrbitStart 主程序：

资源管理、分组、搜索、Command Bar、插件系统



Workspace Plugin：

工作区创建、步骤管理、一键启动、延迟、脚本、窗口恢复

```



近期目标不是完整复刻 PowerToys Workspaces，而是先实现：



```text

创建工作区 → 添加已有资源 → 排序 → 一键按顺序启动

```



然后逐步增强：



```text

启动参数 → 工作目录 → 延迟启动 → 日志 → Command Bar 集成 → 脚本 → 条件等待 → 窗口恢复

```



这样既能满足用户真实需求，又不会让 OrbitStart 主程序变成臃肿的系统工具合集。
